# Client Deliverable Acceptance Criteria Builder

This skill generates clear, mutually agreeable acceptance criteria documents for each project deliverable so clients and solopreneurs share an exact definition of "done" before work begins.

## Acceptance Criteria Workflow

**Step 1 — Gather deliverable details**
Ask the user to list each deliverable in the project (e.g., website homepage, brand logo, API integration). If they've shared a proposal or SOW, extract deliverables from it. For each deliverable, collect:
- What format/medium it will be in (PDF, live URL, code repo, etc.)
- The core function or purpose it must fulfill
- Any known constraints (brand guidelines, word counts, tech stack, platform)
- Number of included revision rounds

**Step 2 — Identify acceptance dimensions**
For each deliverable, build criteria across these dimensions where relevant:
- **Completeness**: What components must be present? (e.g., "Homepage includes hero section, nav, 3 feature blocks, and footer")
- **Functionality**: What must it do or not do? (e.g., "Contact form submits without error on mobile and desktop")
- **Quality standards**: Measurable thresholds (e.g., "Page loads in under 3 seconds," "Copy is free of grammatical errors per Grammarly")
- **Brand/style conformance**: Specific rules (e.g., "Uses only approved hex codes #FFFFFF and #2D2D2D")
- **Deliverable format**: Exact file types, resolution, dimensions, or platform specs
- **Out-of-scope markers**: Explicitly name 2–3 things NOT included to prevent scope creep

**Step 3 — Write criteria as pass/fail statements**
Rewrite each criterion as a binary, verifiable statement the client can check themselves. Use "The deliverable passes when…" framing. Avoid subjective language like "looks professional" or "feels right" — replace with observable specifics.

**Step 4 — Add the revision and sign-off clause**
Append a short clause defining:
- How many revision rounds are included and what qualifies as a revision vs. a new request
- The sign-off method (email confirmation, signed PDF, etc.)
- What happens if criteria are met but client withholds approval

**Step 5 — Format for client presentation**
Present the criteria in plain, non-legal language suitable for pasting directly into a proposal, contract addendum, or onboarding document. Avoid jargon. Use a simple table or numbered list per deliverable.

## Output Format

Produce one section per deliverable using this structure:

---
**Deliverable: [Name]**

| Criterion | Pass Condition |
|---|---|
| Completeness | [Specific components present] |
| Functionality | [What it must do, testable] |
| Quality | [Measurable threshold] |
| Format | [File type, specs, platform] |
| Out of Scope | [2–3 explicit exclusions] |

**Revision Terms:** [X] rounds of revisions included. A revision is defined as [definition]. Additional rounds billed at [rate or TBD].

**Sign-Off Process:** Client confirms acceptance via [email/signed doc] within [X] business days of delivery. Silence after [X] days constitutes acceptance.

---

Repeat for each deliverable. End with a one-paragraph plain-language summary the user can paste into a client email introducing the criteria document. Total length scales with number of deliverables — typically 150–300 words per deliverable.
