# Solo Operator Peak Capacity Response Plan

This skill builds a concrete, ready-to-use protocol for solo operators to triage, respond to, and manage inbound demand when they are at or near full capacity — without losing leads, burning out, or appearing unavailable.

## Capacity Response Protocol Builder

### Step 1: Establish Capacity Baseline
Ask the operator to define (or help them estimate) three numbers:
- **Current load**: How many active clients or projects are in flight right now?
- **True capacity**: What is the realistic maximum (projects, hours/week, revenue)?
- **Buffer threshold**: At what point (e.g., 80% full) should the intake protocol activate?

### Step 2: Classify Inbound Request Types
Sort all inbound requests into four buckets:
1. **Existing clients** — scope additions, urgent requests, new work from current clients
2. **Warm leads** — referred prospects or people who've previously expressed interest
3. **Cold inbound** — unsolicited inquiries, cold outreach, form submissions
4. **Collaboration/partnerships** — non-client work requests (subcontracting, joint ventures)

Each bucket gets a distinct response path, not one generic reply.

### Step 3: Define Response Actions Per Bucket
For each bucket, assign one of these actions:
- **Accept now** — criteria that make something worth squeezing in
- **Hold with date** — send to waitlist with a specific availability date
- **Refer out** — maintain a short list of 2–3 trusted referral partners to redirect to
- **Decline with care** — short, warm, final decline language that preserves the relationship
- **Defer and nurture** — add to a low-touch follow-up sequence for future availability

### Step 4: Write the Holding Response Templates
Create three short message templates (email or DM-ready):
1. **Waitlist message**: Acknowledges interest, gives a specific reopen date, keeps the door open
2. **Referral message**: Declines, offers a named alternative, includes a warm handoff line
3. **Defer message**: For leads worth keeping warm — brief, honest, sets a future check-in date

Each template should be under 100 words, first-person, and sound like the operator — not corporate.

### Step 5: Build the Intake Triage Checklist
A repeatable weekly or trigger-based checklist the operator runs when new requests come in:
- [ ] Is this request within my core service scope?
- [ ] Does this client/project meet my minimum criteria (budget, timeline, fit)?
- [ ] What bucket does this fall into?
- [ ] What is the correct response action?
- [ ] Has the response been sent within 48 hours?

### Step 6: Set a Demand Signal Review Cadence
Define when the operator will review demand volume and adjust thresholds:
- **Weekly**: Are new requests being handled within the protocol?
- **Monthly**: Is the waitlist growing or clearing? Should capacity expand?
- **Trigger-based**: If 3+ referrals happen in one week, flag for capacity reassessment

## Output Format

Produce a single structured document with the following labeled sections:

1. **Capacity Snapshot** — three numbers: current load, true capacity, buffer threshold
2. **Inbound Bucket Definitions** — four buckets with the operator's specific criteria for each
3. **Response Action Map** — a simple table: Bucket | Trigger Condition | Action | Owner (always "me")
4. **Message Templates** — three copy-paste-ready templates (waitlist, referral, defer)
5. **Weekly Triage Checklist** — 5-item checklist formatted as checkboxes
6. **Review Cadence** — bullet list of when
