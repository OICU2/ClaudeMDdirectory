# Beta Offer Feedback Synthesizer

This skill takes raw, unstructured feedback from beta clients or early buyers and converts it into a ranked list of product changes and messaging fixes, so a solopreneur can make confident decisions before a full public launch.

## Feedback Synthesis Workflow

### Step 1: Collect the Raw Material
Ask the user to provide:
- All feedback received (copy-paste, notes, survey responses, DMs — any format)
- The original offer promise or sales page copy (if available)
- Number of beta clients and how they were onboarded
- What outcome the offer was supposed to deliver

If they only have partial info, work with what exists — do not stall.

### Step 2: Categorize Every Piece of Feedback
Sort each feedback item into one of four buckets:

1. **Delivery gaps** — The product/service didn't do what was promised (broken promise, missing step, unclear instruction)
2. **Expectation mismatches** — The client expected something different from what was delivered (messaging problem, not product problem)
3. **Nice-to-haves** — Requests that reflect personal preference, not core promise failure
4. **Validation signals** — What worked, what clients praised, what they'd pay for again

Label each item with its bucket. If one item spans two buckets, split it.

### Step 3: Score Each Issue by Impact
Rate every Delivery gap and Expectation mismatch (ignore nice-to-haves for now) on two axes:

- **Frequency**: How many clients mentioned it? (1 client = low, 2–3 = medium, 4+ = high)
- **Severity**: Would this cause a refund request, bad testimonial, or launch failure? (yes = high severity)

Assign a priority tier:
- **P1 — Fix before launch**: High frequency OR high severity
- **P2 — Fix if time allows**: Medium frequency, low severity
- **P3 — Backlog**: One client, low stakes, personal preference edge

### Step 4: Separate Product Fixes from Messaging Fixes
For each P1 and P2 item, determine:
- **Product fix**: Something must be added, removed, restructured, or clarified inside the offer itself
- **Messaging fix**: The offer is fine but the sales page, onboarding email, or positioning created the wrong expectation

Do not let the user rewrite their whole offer when a one-line messaging clarification would solve it.

### Step 5: Build the Action Plan
Create a concrete fix list:
- For each P1 product fix: state exactly what to change and where
- For each P1 messaging fix: state the specific claim or phrase that caused the mismatch and suggest a replacement
- Flag any pattern in the validation signals that should be amplified in the public launch copy

## Output Format

Produce a structured synthesis report with these sections:

**BETA FEEDBACK SYNTHESIS REPORT**

---

**Offer:** [Name of offer]
**Beta clients:** [Number]
**Feedback items analyzed:** [Number]

---

**CATEGORIZED FEEDBACK TABLE**
A simple table or list grouping each feedback item under its bucket (Delivery Gap / Expectation Mismatch / Nice-to-Have / Validation Signal) with a one-line summary of each item.

---

**PRIORITY FIX LIST**

- [Issue] → [Specific action] → [Product fix or Messaging fix]
