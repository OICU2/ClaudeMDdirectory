# Client Communication Cadence Designer

This skill creates a tiered, scheduled communication system for each client category so the solopreneur controls the rhythm of client interactions instead of reacting to them.

## Cadence Design Workflow

### Step 1: Gather Client Tier Information
Ask the user to list their current clients or client types and sort them into tiers based on:
- Monthly revenue contribution (e.g., high / mid / low)
- Contract type (retainer, project-based, ad-hoc)
- Relationship sensitivity (new, long-term, at-risk)

If they can't segment yet, use these default tiers: **Anchor Clients** (top 20% revenue), **Core Clients** (middle 60%), **Light-Touch Clients** (bottom 20% or one-off).

### Step 2: Assign Communication Channels and Frequency per Tier

**Anchor Clients**
- Weekly async update (written summary: progress, blockers, next steps)
- Bi-weekly live check-in (30 min video or call)
- Monthly strategic review (60 min, agenda sent 48 hrs in advance)
- Response SLA: same business day

**Core Clients**
- Bi-weekly async update
- Monthly live check-in (30 min)
- Quarterly review optional
- Response SLA: within 24 business hours

**Light-Touch Clients**
- Monthly async update or milestone-triggered update only
- No scheduled calls unless client requests
- Response SLA: within 48 business hours

### Step 3: Define the Update Template
For each async update, the user should send a 3-section message:
1. **Done** — what was completed since last update
2. **Next** — what's happening before the next update
3. **Need** — any input, decisions, or assets required from the client

### Step 4: Set Boundary Rules
- All communication routes through one channel per tier (e.g., email for Light-Touch, Slack for Core, dedicated thread for Anchor)
- Define office hours and include them in email signature and onboarding docs
- Ad-hoc requests outside the cadence get a templated response: "Thanks for reaching out — I'll include this in your [day] update. If it's urgent, here's how to flag it: [method]."
- No same-day turnaround promised unless written into contract

### Step 5: Build the Master Calendar Block
Map all recurring touchpoints into a weekly schedule:
- Monday: Draft async updates for the week
- Tuesday/Thursday: Scheduled calls only
- Friday: Send end-of-week summaries to Anchor clients
- Block 90-minute "communication windows" twice daily; no client messages outside those blocks

## Output Format

Produce a **Client Communication Cadence Plan** with the following sections:

1. **Tier Summary Table** — columns: Tier Name, Clients in Tier, Revenue %, Communication Channel, Async Frequency, Live Call Frequency, Response SLA
2. **Weekly Calendar Block** — a plain-text or markdown schedule showing when each communication type happens across the week
3. **Update Message Template** — the Done / Next / Need template filled with placeholder examples relevant to the user's work type
4. **Boundary Response Scripts** — 2–3 copy-paste replies for handling out-of-cadence interruptions (urgent request, quick question, check-in nudge)
5. **30-Day Rollout Checklist** — numbered list of actions to migrate from reactive to scheduled communication, including a note to send a "new communication rhythm" message to all current clients

Total length: 400–600 words. Use markdown headers, tables, and bullet points. No prose paragraphs except in the rollout checklist context notes.
