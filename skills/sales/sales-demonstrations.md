# Sales Demonstrations

This skill guides Claude through planning and delivering compelling product demos that map features to specific prospect problems, keeping the focus on outcomes rather than functionality.

## Demo Workflow

### 1. Gather Prospect Context
Before scripting anything, collect:
- **Company name and industry**
- **Prospect's role and what they care about**
- **Top 2–3 pain points or problems they've expressed**
- **Desired outcome** (what does success look like for them?)
- **Demo format**: live call, recorded video, leave-behind deck, or written walkthrough

### 2. Build the Demo Structure
Follow this sequence for every demo:

**Opening — Anchor to Their Pain (2–3 min)**
- Restate the prospect's problem in their own words
- Confirm the cost of the problem (time, money, risk)
- Set the agenda: "Today I'll show you exactly how we solve X, Y, and Z"

**The Before State (1–2 min)**
- Show or describe what life looks like without the solution
- Make the pain tangible — numbers, manual steps, frustration points

**Feature-to-Outcome Mapping (core of the demo)**
For each key capability, use this structure:
- **Pain**: "You mentioned that [specific problem]..."
- **Show**: Demonstrate the specific feature that addresses it
- **Outcome**: "This means you can now [specific result], which saves/enables/prevents [specific value]"

Limit to 3–5 features maximum. Cut anything that doesn't tie directly to stated pain points.

**Proof Point**
- Insert one customer story or metric: "A [similar company] used this to [result] in [timeframe]"

**Close the Loop (2–3 min)**
- Ask: "Does this solve what you were looking for?"
- Surface objections: "What questions do you have about what you just saw?"
- Define next step before the call ends

### 3. Demo Rules
- Never demo a feature unprompted — only show what connects to their pain
- Speak in outcomes, not features: "You get X" not "We have X"
- Pause after each section and invite reaction
- If prospect goes off-script, follow them — their tangent is a buying signal
- Keep live demos under 20 minutes; leave time for questions

### 4. Handle Common Demo Killers
| Situation | Response |
|-----------|----------|
| "Can you show me [out-of-scope feature]?" | "Absolutely — let me finish this section and we'll go there next" |
| Silence after a feature | "What's your reaction to that?" |
| "We already do that with [competitor]" | "How's that working for you? What would make it better?" |
| Technical issues during live demo | Pivot to recorded clip or screenshots immediately |

## Output Format

Claude will produce a **complete demo script** with the following sections:

1. **Demo Brief** — one paragraph summarizing prospect context, pain points, and goal of this demo
2. **Opening Script** — word-for-word talking points for the first 2–3 minutes
3. **Feature-Outcome Map** — a table with columns: Feature | Prospect Pain It Solves | Outcome Statement to Deliver
4. **Proof Point** — one customer story formatted as: Company type → Problem → Solution used → Result achieved
5. **Closing Script** — questions to ask and how to lock in a next step
6. **Objection Responses** — 3–5 anticipated objections with specific counters

Format: plain text with clear section headers. Length: enough to run a full demo, typically 500–900 words. Write all scripts in first person, ready to speak aloud.
