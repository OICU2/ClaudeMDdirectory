# Proof-of-Value Design

This skill generates a complete, structured POV or pilot plan—including scope, success metrics, timeline, stakeholder roles, and evaluation checkpoints—to give prospects a clear path from evaluation to purchase decision.

## POV Design Workflow

1. **Gather Context First**
   - Ask for: prospect name/industry, primary pain points, solution being evaluated, available time window, key stakeholders, and any known constraints (budget, IT restrictions, data access).
   - If context is partially provided, infer reasonable defaults and flag assumptions clearly.

2. **Define the Business Problem**
   - State the specific pain point or opportunity the POV addresses in one sentence.
   - Tie it to a measurable business outcome (e.g., reduce churn, increase throughput, cut processing time).

3. **Set Success Criteria**
   - Define 3–5 quantifiable success metrics tied to the stated problem.
   - Include: baseline value (current state), target value (desired state), and measurement method.
   - Separate "must-have" metrics (deal-critical) from "nice-to-have" metrics (supporting evidence).

4. **Scope the POV**
   - Define what is IN scope: specific use cases, user groups, data sets, integrations, or workflows to be tested.
   - Define what is OUT of scope to prevent scope creep and set clear expectations.
   - Specify minimum viable requirements to declare the POV valid.

5. **Build the Timeline**
   - Propose a phased timeline (typically 2–6 weeks) with named phases:
     - **Setup & Onboarding** (Days 1–X): environment config, data access, stakeholder kickoff.
     - **Active Evaluation** (Days X–Y): core usage, data collection, issue resolution.
     - **Analysis & Review** (Days Y–Z): results compilation, stakeholder readout, decision.
   - Flag dependencies that could delay each phase.

6. **Assign Stakeholder Roles**
   - List required roles on both sides: Champion, Technical Lead, Executive Sponsor, Day-to-Day User.
   - For each role, specify their responsibility during the POV (not just their title).

7. **Define Evaluation Checkpoints**
   - Week 1: Setup confirmation and baseline measurement locked.
   - Midpoint: Progress check against success metrics, issue triage.
   - Final: Results readout with go/no-go recommendation criteria stated in advance.

8. **Identify Risk Factors**
   - List 2–4 risks that could invalidate the POV (data not available, wrong users involved, limited usage).
   - Provide a mitigation action for each.

9. **State the Decision Framework**
   - Define the exact conditions under which the prospect should say yes, no, or needs more time.
   - Include a proposed next-step meeting on the calendar before the POV ends.

## Output Format

Produce a structured POV plan document with the following sections in order:

**Header:** POV title, prospect name, solution name, POV owner, date, duration.

**1. Business Problem** — 2–3 sentences.

**2. Success Metrics Table** — Columns: Metric | Baseline | Target | Measurement Method | Priority (Must-Have / Nice-to-Have).

**3. Scope** — Two bullet lists: In Scope / Out of Scope.

**4. Timeline** — Phased table or numbered list with dates, milestones, and owners.

**5. Stakeholder Responsibilities** — Table with columns: Role | Name | Responsibility.

**6. Evaluation Checkpoints** — Numbered list with date, attendees, agenda, and output for each checkpoint.

**7. Risk Register** — Table with columns: Risk | Likelihood | Impact |
