# Pilot Program Proposal Design

This skill builds a complete, time-boxed pilot proposal with defined success criteria, scope boundaries, timeline, and a clear escalation path to a full contract — used when prospects need a lower-commitment entry point to move forward.

## Pilot Proposal Design Workflow

### 1. Gather Context Before Building
Ask for (or extract from conversation):
- What the prospect's core hesitation is (budget, risk, internal approval, technical fit)
- Which specific use case or problem they want to validate
- Their stakeholders and who defines success internally
- Any hard constraints: timeline, budget ceiling, team availability, or integration limits

### 2. Define the Pilot Scope
Set explicit boundaries to prevent scope creep:
- **In scope**: List 2–4 specific deliverables or features being tested
- **Out of scope**: Name at least 2 things explicitly excluded
- **User group**: Define the subset of users, teams, or data involved (never "everyone")
- **Environment**: Specify sandbox, staging, or limited production

### 3. Set Measurable Success Criteria
Define 3–5 binary or quantifiable outcomes the prospect must agree to upfront:
- Use the prospect's language and metrics where possible (e.g., "reduce ticket resolution time by 20%")
- Include at least one leading indicator (early signal) and one lagging indicator (final outcome)
- Assign ownership: who measures each metric and how
- State the threshold: what "pass" looks like vs. "fail"

### 4. Build the Timeline
Structure a 30, 60, or 90-day arc with named milestones:
- **Week 1–2**: Onboarding, configuration, baseline measurement
- **Midpoint check-in**: Progress review against success criteria, flag blockers
- **Final week**: Measurement, readout, and go/no-go decision meeting
- Include a fixed end date — open-ended pilots stall

### 5. Define the Escalation Path
Make the transition to full contract frictionless:
- State what a successful pilot automatically unlocks (pricing, terms, scope)
- Pre-agree on the decision meeting date and attendees
- Include a soft commitment clause: "If criteria X and Y are met, we proceed to full contract by [date]"
- Identify the internal champion who will own the go/no-go recommendation

### 6. Address Risk Mitigation
Reduce perceived risk explicitly:
- State what the prospect is NOT committing to beyond the pilot
- Include a clear exit clause if criteria are not met
- Cap resource requirements on the prospect's side (hours/week from their team)
- Specify data handling, security, or compliance boundaries if relevant

## Output Format

Produce a structured pilot proposal document with the following sections:

**Pilot Program Proposal: [Prospect Name] × [Your Company]**

1. **Objective** — 2–3 sentences on what the pilot is designed to prove
2. **Scope** — Bulleted in-scope and out-of-scope lists
3. **Success Criteria** — Table with: Metric | Target | Owner | Measurement Method
4. **Timeline** — Visual or bulleted milestone schedule with dates
5. **Resource Requirements** — What each party commits (hours, personnel, access)
6. **Escalation Path** — Conditions and timeline for full contract conversion
7. **Exit Clause** — What happens if criteria are not met
8. **Next Steps** — 3 numbered actions with owners and due dates

Length: 400–700 words. Tone: direct and professional. Format: ready to send as a proposal document or paste into a deck slide. Avoid legal language — keep it readable by non-lawyers.
