# Product Feature Announcement

This skill generates a complete, coordinated launch brief for a new product feature, producing ready-to-use copy for email, in-app messaging, social media, and a changelog entry — all aligned in tone and messaging.

## Launch Brief Workflow

1. **Gather feature context** — Before writing, confirm or ask for:
   - Feature name and one-line description
   - Primary user benefit (what problem does it solve?)
   - Target audience (all users, specific plan tiers, new vs. existing users)
   - Launch date or timing
   - Any beta/early access context, limitations, or prerequisites
   - Desired tone (e.g., excited, understated, technical, friendly)

2. **Identify the core message** — Distill the feature into a single value proposition sentence. Every channel asset must reinforce this message. Lead with the user benefit, not the technical implementation.

3. **Write each channel asset** using the specs below:

   **Email announcement**
   - Subject line: 6–9 words, benefit-led, no clickbait
   - Preview text: 1 sentence, complements subject line
   - Body: 150–250 words. Open with the problem the feature solves. Introduce the feature by name. List 2–3 specific benefits as bullets. Close with a single, clear CTA button label and destination description.

   **In-app message / tooltip / modal**
   - Headline: 5–8 words max
   - Body: 1–2 sentences (40–60 words). Focus on immediate action the user can take right now.
   - CTA: 2–4 word button label

   **Social posts**
   - Write 3 variants:
     - LinkedIn (100–150 words, professional tone, include a hook line and 3–5 relevant hashtags)
     - Twitter/X (under 240 characters, punchy, optional emoji, 1–2 hashtags)
     - Optional long-form Twitter/X thread opener (2–3 tweets) if the feature warrants explanation

   **Changelog entry**
   - Title: "[Feature Name] — [Date or Version]"
   - Type label: New Feature / Improvement / Fix
   - Body: 3–5 sentences in plain, factual language. Describe what changed, why, and how to access it. Avoid marketing language — changelog readers are power users.

4. **Flag inconsistencies** — If any asset contradicts another in tone, claim, or CTA, note it explicitly and suggest a fix.

5. **Offer variations on request** — After delivering the full brief, offer to adjust tone, rewrite for a specific audience segment, or create a shortened version for any channel.
