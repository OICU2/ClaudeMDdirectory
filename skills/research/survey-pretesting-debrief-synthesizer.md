# Survey Pretesting Debrief Synthesizer

This skill processes raw cognitive interview or think-aloud debrief notes from survey pre-tests and produces a prioritized, structured list of item revisions with evidence-based rationale.

## Synthesis Workflow

1. **Intake and parse notes**: Accept notes in any format (bullet points, transcripts, interviewer summaries). Identify which survey item each comment references. If item labels are missing, ask the user to clarify before proceeding.

2. **Classify each problem by type**: Tag each observed issue using standard cognitive interview problem categories:
   - *Comprehension* — respondent misunderstood the question wording or intent
   - *Retrieval* — respondent had difficulty recalling the required information
   - *Judgment/Estimation* — respondent struggled to form or calculate an answer
   - *Response mapping* — respondent could not match their answer to the provided options
   - *Sensitivity/Motivation* — respondent was unwilling to answer honestly or at all
   - *Instructions/Format* — layout, skip logic, or administrative text caused confusion

3. **Assess severity and frequency**: For each flagged item, note how many respondents raised the issue and whether the problem would produce systematic bias, random error, or skip/dropout. Classify severity as High (affects validity or completion), Medium (degrades precision), or Low (minor friction).

4. **Draft specific revisions**: For every High and Medium severity issue, propose a concrete revision — reworded question text, revised response options, added clarifying instruction, or structural change. Do not recommend vague improvements like "clarify wording." Write the actual proposed new text.

5. **Preserve unchanged items**: Explicitly list items that generated no problems so stakeholders know which items cleared pretesting.

6. **Flag unresolved issues**: If a problem has no clear fix, or if competing revisions would trade one error type for another, flag it as "Needs Expert Review" with a brief explanation.

## Output Format

Produce a structured report with the following sections:

**Summary Table** — one row per flagged item, columns: Item ID | Problem Type | Severity | # Respondents Affected | Recommended Action (Revise / Flag / No Change)

**Revision Details** — for each item requiring revision:
- *Item ID and original text*
- *Problem observed* (1–2 sentences citing the note evidence)
- *Problem type and severity*
- *Proposed revision* (full replacement text, clearly marked)
- *Rationale* (one sentence explaining why the change addresses the root cause)

**Items Passing Pretest** — bulleted list of item IDs with one-line confirmation

**Flagged for Expert Review** — bulleted list of items with unresolved conflicts, each with a 1–2 sentence description of the dilemma

Target length: as long as the item count requires; do not truncate revision details for brevity. Use plain language throughout so non-methodologist stakeholders can act on the report directly.
