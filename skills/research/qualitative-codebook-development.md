# Qualitative Codebook Development

This skill builds a rigorous, structured codebook that enables consistent qualitative coding across researchers, datasets, and analytical sessions.

## Codebook Development Workflow

1. **Clarify the research context** before building anything:
   - What is the research question or phenomenon under study?
   - What type of data is being coded (interviews, documents, observations, social media)?
   - What coding approach applies (deductive, inductive, or hybrid)?
   - Are codes already partially identified, or does the codebook need to be built from scratch?

2. **Establish the code inventory** by doing one of the following:
   - If codes are provided: work with the supplied list
   - If a theoretical framework is provided: derive codes from it
   - If raw excerpts are provided: generate candidate codes inductively
   - If only a topic is provided: generate a starter set of codes with a disclaimer that they require researcher review

3. **For each code, develop all required components:**
   - **Code Name**: Short, descriptive label (2–5 words, no jargon unless field-standard)
   - **Code ID**: Alphanumeric identifier (e.g., EMO-01, SOC-03) for tracking
   - **Definition**: 2–4 sentence conceptual definition explaining what the code captures and why it matters theoretically or analytically
   - **Inclusion Criteria**: Explicit rules for when to apply the code — what must be present in the data (topics, language, meaning, context)
   - **Exclusion Criteria**: Explicit rules for when NOT to apply the code — common confusions, boundary cases, and what distinguishes this code from adjacent ones
   - **Prototypical Example**: A clear, concrete excerpt that is a textbook instance of the code, with a brief annotation explaining why it qualifies
   - **Borderline Example**: An ambiguous excerpt that illustrates a gray area, with guidance on how to decide
   - **Related Codes**: List of codes that overlap or are commonly confused with this one, with a note on how to distinguish them

4. **Organize codes into a hierarchy if applicable:**
   - Group codes into parent themes or domains
   - Label each level clearly (Theme → Category → Code → Sub-code)
   - Note whether sub-codes are mutually exclusive or can co-occur

5. **Add codebook-level documentation:**
   - Coding unit definition (what counts as one unit: sentence, turn, paragraph, passage?)
   - Rules for applying multiple codes to a single unit
   - Instructions for handling ambiguous or uncodeable passages
   - Version number and date

6. **Flag gaps or concerns:**
   - Note any codes that are vague, overlapping, or likely to cause low intercoder agreement
   - Suggest pilot testing on a small data sample before full deployment

## Output Format

Produce the codebook as a structured document with the following layout:

---

**CODEBOOK: [Research Project Title or Topic]**
**Version:** 1.0 | **Date:** [today's date] | **Coding Unit:** [defined unit]

---

**GENERAL CODING RULES**
- [Bullet list of 3–6 project-level rules]

---

**THEME 1: [Theme Name]**

| Field | Content |
|---|---|
| **Code ID** | [e.g., THM1-01] |
| **Code Name** | [Short Label] |
| **Definition** | [2–4 sentences] |
| **Inclusion Criteria** | [Bulleted list of what qualifies] |
| **Exclusion Criteria** | [Bulleted list of what does not qualify] |
| **Prototypical Example** | *"[excerpt]"* — [1–2 sentence annotation] |
| **Borderline Example** | *"[excerpt]"* — [guidance on how to decide] |
|
