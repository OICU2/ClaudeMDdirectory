# Research Scope Extension Impact Brief

This skill produces a structured impact brief estimating the time, cost, and data implications of expanding a research scope, enabling stakeholders to make informed go/no-go decisions on scope changes.

## Scope Extension Analysis Workflow

1. **Clarify the baseline scope** — Identify the current research boundaries: sample size, data sources, variables, population, timeline, and budget. Ask the user to confirm these if not already stated.

2. **Define the proposed extension** — Precisely articulate what is being added: additional cohorts, variables, geographic regions, data collection waves, methodologies, or analysis layers.

3. **Estimate incremental time** — Break down added time by phase: recruitment/data collection, processing/cleaning, analysis, and reporting. Provide a range (optimistic / realistic / pessimistic).

4. **Estimate incremental cost** — Itemize cost drivers introduced by the extension: personnel hours, participant incentives, licensing or data acquisition, tooling, and overhead. Flag assumptions explicitly.

5. **Identify data requirements** — List new data sources, minimum sample sizes, access requirements, consent considerations, and any data quality risks specific to the extension.

6. **Assess feasibility risks** — Note constraints that could make the extension impractical: recruitment difficulty, data availability, regulatory hurdles, or skill gaps.

7. **Document trade-offs** — For each extension option, state what is gained (statistical power, generalizability, stakeholder coverage) versus what is sacrificed (speed, budget, simplicity, focus).

8. **Present decision options** — Offer 2–3 scoping tiers (e.g., minimal extension, moderate extension, full extension) so stakeholders can compare alternatives rather than evaluate a single proposal.

## Output Format

Produce a brief structured document with the following sections:

**Scope Extension Impact Brief**

- **Current Scope Summary** (2–3 bullet points covering sample, timeline, budget baseline)
- **Proposed Extension** (1 short paragraph describing exactly what would be added)
- **Impact Estimates Table**

| Dimension | Current Scope | Extension Option A | Extension Option B |
|---|---|---|---|
| Timeline | | | |
| Estimated Cost | | | |
| Sample / Data Size | | | |
| Key New Requirements | | | |

- **Feasibility Risks** (bulleted list, 3–6 items)
- **Trade-Off Summary** (two columns: What You Gain / What You Give Up, 3–5 rows)
- **Recommendation** (1–2 sentences identifying the option that best balances rigor and practicality, with a confidence qualifier if data is limited)
- **Open Questions for Stakeholders** (2–4 questions that must be answered before committing to an extension)

Length: 400–700 words excluding tables. Use plain language suitable for non-technical sponsors. Flag any estimates based on assumptions rather than confirmed data with an asterisk and footnote.
