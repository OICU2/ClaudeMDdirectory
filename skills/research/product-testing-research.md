# Product Testing Research

This skill designs end-to-end product testing plans and synthesizes findings into prioritized, launch-ready insights when someone needs to validate a product, feature, or prototype before release.

## Testing Research Workflow

### 1. Define Testing Objectives
- Clarify what decisions the test results must inform (launch/no-launch, feature prioritization, pricing, UX changes)
- Identify the riskiest assumptions: usability, desirability, technical reliability, or market fit
- Set a clear success threshold (e.g., "70% of testers complete core flow without assistance")

### 2. Select Testing Method(s)
Match method to objective:
- **Usability testing** — observe real users attempting specific tasks; use for UX and flow validation
- **Beta/dogfooding** — release to limited real users; use for reliability and retention signals
- **Concept testing** — show mockups or descriptions before building; use to validate desirability early
- **A/B or prototype testing** — compare variants; use when specific design decisions are in dispute
- **Diary/longitudinal studies** — participants log usage over days/weeks; use for habit-forming products

### 3. Define Participant Criteria
- Specify 3–5 screener criteria matching your target customer profile
- Determine sample size: minimum 5 per segment for qualitative; 50+ per variant for quantitative
- Identify recruitment channel: existing waitlist, panel services (UserTesting, Prolific), or direct outreach

### 4. Build the Test Protocol
- Write a scenario-based task list (not leading questions — frame as real situations)
- Prepare a moderation guide: opening script, probing questions, closing debrief
- Define metrics to capture: task completion rate, time-on-task, error rate, satisfaction score (CSAT or SUS), qualitative themes

### 5. Run the Test
- Pilot with 1–2 internal participants to catch protocol gaps
- Collect both behavioral data (what they do) and attitudinal data (what they say)
- Record sessions with consent; take timestamped notes on critical moments

### 6. Synthesize Findings
- Cluster observations into themes using affinity mapping
- Rank issues by frequency × severity (P0: blocks launch, P1: degrades experience, P2: minor friction)
- Map findings back to original assumptions — confirm, refute, or flag as inconclusive

### 7. Produce Recommendations
- For each P0/P1 issue: state the problem, the evidence, and a concrete fix
- Provide a launch readiness verdict: Ready / Ready with conditions / Not ready
- Flag what to monitor post-launch if gaps remain

## Output Format

Produce a **Product Testing Research Report** with these sections:

**1. Testing Overview**
- Objective, method(s) used, participant count and profile, dates

**2. Test Protocol Summary**
- Key tasks, metrics tracked, success criteria

**3. Findings by Severity**
- Table format: Issue | Frequency | Severity | Evidence | Recommendation
- P0, P1, P2 grouped separately

**4. Assumption Validation Summary**
- Bullet list: each assumption → Confirmed / Refuted / Inconclusive + one-line rationale

**5. Launch Readiness Verdict**
- Clear verdict with 2–4 sentence justification
- Conditions or watch items if launching with known gaps

**6. Next Steps**
- Numbered action list with owners and suggested timelines

Length: Concise executive summary (½ page) + full detail sections. Use tables and bullets throughout. Avoid narrative paragraphs except in the verdict section.
